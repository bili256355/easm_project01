# UPDATE_LOG_V6_1.md

## 版本定位
`V6_1` 定位为：**依赖 V6 点层结果的窗口派生子版本**。  
职责是：

1. 从 `V6` 的点层结果派生窗口对象  
2. 给窗口主中心附加 bootstrap 返回区间/定位不确定性描述

`V6_1` 不承担：

- 第二套点层主显著性裁决
- competition
- parameter-path
- final judgement
- yearwise gating

## 关键更新

### 1. 版本治理方式修正
`V6_1` 已改为**独立子版本代码树**，而不是继续挂在 `V6` 本体内。  
当前理解为：

- `V6`：主版本，做点层主筛
- `V6_1`：子版本，读取 `V6` 结果并派生窗口层

`V6_1` 可以：

- 接收 `V6` 输出结果
- 复用 `V6` 稳定底座接口

但保持：

- 独立入口
- 独立配置
- 独立 pipeline
- 独立 outputs/logs

### 2. 初版窗口层暴露出严重错误
初版 `V6_1` 中曾出现：

- `CP005 (113) -> 66–119`

并错误地把 `81` 和 `113` 并进同一窗口。  
已确认原因有两层：

- **设计问题**：band 构造过宽，没有尊重相邻独立峰边界
- **实现问题**：merge 逻辑错误使用“最后一个 band 的 end_day”而不是“当前组最大 end_day”

### 3. 窗口层已完成第一轮修复
本轮修复后，主要改动为：

- band 构造更保守
- 加入峰间候选点/局部谷值截断
- 加入最大 band 半宽限制
- 修复 merge 的组右边界 bug
- 对已过点层主筛的主点增加保护，不再轻易被误并窗
- 近邻点仍允许共享窗口，因此 `132/135` 仍可自然合窗

### 4. 当前窗口结果已明显合理化
当前已确认：

- `45` 单独成窗
- `81` 单独成窗
- `113` 单独成窗
- `160` 单独成窗
- `132 / 135` 共享一个窗口
- 不确定性层仍然只是窗口属性，没有长成第二套窗口对象

也就是说，`81` 与 `113` 的错误合并已经被修掉。

### 5. 当前窗口主线使用建议
后续路径层当前优先使用的窗口对象为：

- **45 对应窗口**
- **81 对应窗口**
- **113 对应窗口**
- **160 对应窗口**

`132 / 135` 的共享窗当前先保留记录，暂不作为主路径层重点对象。  
其主峰是 `132` 还是 `135`，当前不影响主线推进，后续需要时再回看。

### 6. 五阶段使用原则
当前后续路径层可采用：

- **4 个已过显著性的转换窗口**
- 以及**由这些转换窗口切分出的 5 个阶段**

作为后续时间对象层。

这里的关键是：

- 主线先围绕四个过显著性的转换窗口推进
- 五阶段是由这些窗口切分出来的阶段对象
- 不再让 `132 / 135` 这类未过主显著性的近邻对象干扰主线

## 当前主线结论
`V6_1` 当前已经基本达到“可用于后续路径层时间对象层”的程度。  
当前可先定住的主结构是：

- 四个已过显著性的转换窗口
- 以及由其切分出的五个阶段
